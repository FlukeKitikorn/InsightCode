# AI Analytics: สิ่งที่ LLM วิเคราะห์ได้จริง และการออกแบบหน้า

## 1. สิ่งที่ LLM สามารถวิเคราะห์ได้จริง (Feasible)

| รายการ | คำอธิบาย | ข้อมูลที่ใช้ | ความเป็นไปได้ |
|--------|----------|--------------|----------------|
| **วิเคราะห์โค้ดต่อ submission** | คุณภาพ โครงสร้าง ความซับซ้อน คำแนะนำปรับปรุง | `code`, `language`, ผลรัน (passed/failed) | ✅ ทำอยู่แล้ว (rule-based) หรือเปลี่ยนเป็น OpenRouter ได้ |
| **สรุปแนวโน้มจากหลาย submission** | เช่น "คุณมักใช้ nested loop ลองใช้ Map" | หลาย `AiFeedback.analysisText` + สถิติ | ✅ ส่ง context สรุปให้ LLM แล้วให้สรุปเป็นข้อความสั้น |
| **ตอบคำถามแบบอิสระ (Ask AI)** | "ควรฝึกอะไรต่อ?" "จุดอ่อนของฉันคืออะไร?" | สถิติ progress + ข้อความ feedback ล่าสุด | ✅ ส่ง JSON สรุปให้ LLM แล้วให้ตอบเป็นข้อความ |
| **ให้คะแนนคุณภาพ (Quality Score)** | คะแนน 0–100 ต่อ submission | โค้ด + ภาษา | ✅ ทำได้ทั้ง rule-based และ LLM |

## 2. สิ่งที่ LLM วิเคราะห์ไม่ได้จากข้อมูลที่มี (ไม่ควรอ้าง)

ระบบเรา **ไม่มี** ข้อมูลเหล่านี้ จึงไม่ควรแสดงเป็น "ผลวิเคราะห์ AI":

| รายการ | เหตุผล |
|--------|--------|
| Typing cadence / ความเร็วพิมพ์ | ไม่มีการเก็บ keystroke หรือ timing ต่อตัวอักษร |
| Mental state / Flow / Anxious | ไม่มี biosignal หรือ behavioral sensor |
| Time spent per function | ไม่มี IDE integration วัดเวลาต่อฟังก์ชัน |
| "Focus Score" แบบ real-time | ไม่มีข้อมูลการโฟกัสจริง |

ดังนั้น **หน้า AI Analytics ควรเน้นเฉพาะสิ่งที่ได้จาก (1) ข้อมูลใน DB และ (2) ผลที่ LLM/rule-based วิเคราะห์จากโค้ดจริง**

---

## 3. ออกแบบหน้า AI Analytics ให้สอดคล้อง

### 3.1 โครงสร้างหน้า (สอดคล้องกับข้อมูลจริง + สิ่งที่ LLM ทำได้)

1. **สรุปความคืบหน้า (จาก API จริง)**  
   - ใช้ `GET /api/users/me/progress`  
   - แสดง: โจทย์ที่ทำแล้ว/ที่ผ่าน, Mastery ต่อความยาก (EASY/MEDIUM/HARD)  
   - **ไม่ใช้ LLM** แค่แสดงสถิติจาก DB  

2. **แนวโน้มคะแนนคุณภาพจาก AI (Quality Score)**  
   - ใช้ข้อมูลจาก `AiFeedback.qualityScore` ของ user คนนั้น  
   - แสดงเป็นกราฟหรือค่าเฉลี่ย (เช่น 7 วันล่าสุด)  
   - **ที่มา:** LLM หรือ rule-based วิเคราะห์จากโค้ดที่ส่ง  

3. **Recent AI Insights (ผลวิเคราะห์ล่าสุด)**  
   - แสดงรายการ submission ล่าสุดที่มี `AiFeedback`  
   - แต่ละรายการ: โจทย์, วันที่, สถานะ (accepted/wrong), ข้อความสรุปจาก `analysisText` (หรือประโยคสั้นจาก LLM)  
   - **ที่มา:** มาจาก LLM/rule-based ต่อ submission  

4. **Ask AI (ถ้าต้องการให้ LLM วิเคราะห์เพิ่ม)**  
   - ช่องข้อความ: ผู้ใช้พิมพ์คำถาม เช่น "ควรฝึกอะไรต่อ?"  
   - Backend รวบรวม context: สถิติ progress + ข้อความ feedback ล่าสุด (หรือสรุปจาก `analysisText`)  
   - ส่งไป OpenRouter → ได้ข้อความตอบกลับมาแสดง  
   - **ที่มา:** LLM วิเคราะห์จาก context ที่ส่งไป  

### 3.2 สิ่งที่ไม่ควรมีในหน้า (เพื่อไม่ให้อ้างเกินจริง)

- Focus Score / Mental State / Flow ที่อ้างว่าเป็น "ผลวิเคราะห์พฤติกรรมจริง"  
- Time per function / Typing cadence ที่ไม่มีข้อมูลจากระบบ  

ถ้าต้องการเก็บ "ความรู้สึก" หรือ "ช่วงโฟกัส" ไว้แสดง ต้องระบุชัดว่าเป็น **ตัวอย่างหรือ placeholder** ไม่ใช่ผลจาก AI จริง

---

## 4. สรุป

- **LLM วิเคราะห์ได้จริง:** โค้ดต่อข้อ, สรุปแนวโน้มจากหลาย feedback, ตอบคำถามจาก context (progress + feedback), ให้คะแนนคุณภาพ  
- **ออกแบบหน้า AI Analysis ให้:** ใช้ข้อมูลจาก DB + ผลที่วิเคราะห์จากโค้ด (rule-based หรือ LLM) เท่านั้น และแยกส่วน "Ask AI" ถ้ามีการเรียก OpenRouter  
- **ไม่ควรอ้าง:** พฤติกรรมการพิมพ์, สภาวะจิต, เวลาต่อฟังก์ชัน ถ้าไม่มีข้อมูลจริงในระบบ  

ไฟล์นี้ใช้เป็น reference สำหรับการ implement หน้า AiAnalyticsPage และ API ที่เกี่ยวข้อง (เช่น `GET /api/users/me/insights` สำหรับ recent submissions + AI feedback).
